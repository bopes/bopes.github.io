Dynamic vs Static Languages

Mar 14 2016

All coding languages are either dynamically or statically typed. The difference is determined by how that language handles variables.

A dynamic language lets variables be redefined to any type at any given point - the variable type is dynamic. Ruby and JavaScript both fit this description. Here's how it would look:

num = 10
 num = "ten"

Here you can see that the first line sets the variable equal to an integer, but the second sets it equal to a string.

In static languages, this isn't allowed. You must declare your variable and variable type before defining it, and the variable must equal that datatype. If it is set to another type, it will return an error.

There are advantages and disadvantages to both types.

Dynamic languages are faster to type and produce the most readable code. However, by not enforcing variable data types, it is possible to accidentally assign incorrect values to variables or create new, mistyped variables on the fly. To fix these errors you have to run the code and find any runtime errors. This testing takes longer and is harder to do, resulting in more errors slipping through to production. Generally speaking, the benefits of dynamic languages make them better suiting for prototyping and shorter development cycles.

On the other hand, static languages can find these errors for you when you compile the code. This makes testing considerably easier. The downside is that it takes longer write, as you have to declare your variables in advance. These languages are better for larger, more precise applications, as they have more robust testing capabilities and better system performance.

Because of the best practices found in each, it is typically easier for experienced static-language coders to switch to dynamic-languages than vice versa. But for your application, you should start with whatever language you know best, and then go from there.
